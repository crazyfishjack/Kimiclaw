## 1. 项目概述

AI 晨报推送机器人是一个自动化新闻聚合与推送系统，专为关注 AI 领域动态的用户设计。系统每日自动抓取多个高质量技术源的内容，生成中文结构化摘要，并通过微信推送给用户，同时自动存档到 GitHub。

### 1.1 核心目标

- **自动化**: 无需人工干预，每日自动抓取、总结、推送
- **高质量**: 基于多源数据，经过质量筛选和去重，只推送精品内容
- **中文优先**: 所有输出强制为中文结构化格式
- **可追溯**: 所有推送内容自动存档到 GitHub，支持历史查阅

## 2. 需求来源

### 2.1 用户原始需求（对话记录整理）

**数据源要求:**

| 类别 | 具体来源 |
|------|---------|
| 国际论文 | arXiv（cs.AI / cs.CL / cs.CV / cs.LG） |
| 开源项目 | GitHub Trending（日榜/周榜） |
| 国内大厂 | 字节跳动技术团队、阿里技术、腾讯技术工程、百度AI |
| 技术媒体 | 机器之心、量子位、AI科技评论、新智元、InfoQ、CSDN |
| 开发者社区 | 稀土掘金、知乎 AI 话题 |
| 垂直公众号 | 新智元、机器之心、量子位、数字生命卡兹克、刘小排、磐创 AI、AI 有道、深度学习初学者、极客公园 |

**推送要求:**

1. 每天早上 8 点自动推送一个合集
2. 用户可随时通过对话触发实时推送（如发送"推送晨报"）
3. 推送消息需包含摘要和原文链接
4. 内容必须是中文结构化摘要，不能是简短几句话

**存档要求:**

1. 每次推送同步存档到 GitHub
2. 文件名为中文日期格式：`YYYY年MM月DD日_AI晨报.md`
3. 存档内容必须包含详细中文摘要，不能只是英文摘要
4. 摘要格式统一为：【核心问题】【方法】【结果/意义】

### 2.2 演化需求

在构建过程中，用户逐步明确了以下关键要求：

1. **深度总结**: 不能是模板化填空，必须基于原文内容详细分析后生成中文摘要
2. **纯中文输出**: 所有内容（包括论文、项目、文章）必须全部使用中文，不能混英文
3. **永久去重**: 同一内容一旦推送过，永远不再重复推送
4. **GitHub 同步**: 推送微信消息的同时，必须同步更新 GitHub 存档

## 4. 核心模块设计

### 4.1 数据抓取模块 (`fetchers/`)

#### arXiv 抓取器

- **API**: arXiv API (http://export.arxiv.org/api/query)
- **分类**: cs.AI, cs.CL, cs.CV, cs.LG
- **抓取数量**: 每类 5 篇，共约 20 篇
- **字段**: title, summary, authors, published, arxiv_id, url
- **频率**: 每日一次

#### GitHub Trending 抓取器

- **源**: GitHub Trending 页面 (非官方 API)
- **筛选条件**: 语言不限，按 Stars 数排序
- **抓取数量**: 每日 10-16 个
- **字段**: name, description, language, stars, url

#### 稀土掘金抓取器

- **API**: 稀土掘金官方 API
- **标签**: AI、人工智能、大模型
- **抓取数量**: 10 篇
- **字段**: title, content, author, url, stars

#### 微信公众号 RSS 抓取器

- **源**: 各公众号 RSS 订阅源
- **当前状态**: RSS 服务不稳定（频繁返回 HTTP 503）
- **待优化**: 需要寻找更稳定的 RSS 聚合服务或自建代理

### 4.2 质量筛选模块 (`quality_filter.py`)

**评分维度:**

| 维度 | 权重 | 说明 |
|------|------|------|
| 来源权威性 | 30% | arXiv > GitHub > 掘金 > 公众号 |
| 内容完整度 | 25% | 标题+摘要+链接完整 |
| 时效性 | 20% | 发布日期越近分数越高 |
| 互动数据 | 15% | Stars、点赞数等 |
| 关键词匹配 | 10% | 标题含 AI/ML/LLM 等关键词 |

**筛选阈值**: 质量分 ≥ 45 分才进入候选池

### 4.3 去重模块 (`deduplicator.py`)

**去重规则:**

- **arXiv**: 按 `arxiv_id` 去重
- **GitHub**: 按 `repo_name` 去重
- **掘金/文章**: 按 `url` 去重
- **通用 fallback**: 按 `title_hash` 去重

**存储**: `/tmp/news_sent_history.json`  
**保留期**: 永久（730 天自动清理过期记录）

**注意**: 永久去重意味着同一内容一旦推送过，永远不会再次出现。这在保证新鲜度的同时，可能导致每天可推送的内容逐渐减少。

### 4.4 中文结构化总结模块 (`llm_summarizer.py`)

**核心设计原则:**

1. **纯中文输出**: 所有内容必须翻译成中文，不能保留英文原文
2. **基于原文分析**: 不是模板填空，而是先分析原文句子结构，再生成中文
3. **三段式结构**:
   - 【核心问题】2-3 句话说明研究/项目要解决什么问题
   - 【方法】2-3 句话说明用了什么技术/方案
   - 【结果/意义】2-3 句话说明取得了什么效果

**处理流程:**

```
输入: title + abstract/description
  ↓
句子功能分类 (背景/方法/结果)
  ↓
关键词匹配 → 选择对应的中文描述模板
  ↓
生成结构化中文摘要
  ↓
输出: 【核心问题】...【方法】...【结果/意义】...
```

**类型分发:**

- **论文** (`arxiv.org/abs` 链接): 使用 `_analyze_sentences()` 分析 abstract，按关键词生成中文
- **GitHub 项目**: 使用 `_github_summary()`，基于 description 关键词分类（教程/工具/框架等）
- **中文文章**: 使用 `_article_summary()`，基于标题关键词分类（工具对比/趋势分析/教程等）

**关键关键词映射:**

| 类型 | 关键词 | 生成内容 |
|------|--------|---------|
| 论文-多模态 | multimodal, visual understanding | 多模态理解与生成割裂问题 |
| 论文-强化学习 | reinforcement learning, RLVR | 无需标准答案的 RL 训练 |
| 论文-扩散模型 | diffusion, editing | 图像生成统一编辑能力 |
| 论文-注意力 | attention, noise, softmax | 注意力噪声抑制 |
| 项目-教程 | tutorial, course, hub | 降低 AI 工程学习门槛 |
| 项目-API网关 | api, gateway, hub | 统一多模型接入 |
| 项目-记忆 | memory, knowledge graph | Agent 长期记忆能力 |
| 文章-工具对比 | cursor, codex, 感受 | AI 编程工具对比体验 |
| 文章-排行榜 | github, 排行, 热门 | 技术趋势盘点分析 |

## 5. 推送机制

### 5.1 自动推送 (HEARTBEAT)

**触发条件:** 每次用户与 bot 交互时

**执行逻辑:**

```python
if /tmp/morning_news.json 存在且是今天的:
    读取内容 → 生成摘要 → 推送微信 → GitHub存档 → 删除文件
elif 今天还没推送过:
    实时抓取 → 生成摘要 → 推送微信 → GitHub存档
else:
    发送 HEARTBEAT_OK (静默)
```

**问题**: HEARTBEAT 只在用户发消息时触发。如果用户早上 8 点没发消息，晨报不会自动推送。

**解决方案**: 系统 cron 每日 7:50 预抓取数据保存到 `/tmp/morning_news.json`，用户任何时间发消息都能触发推送。

### 5.2 手动触发

用户发送"推送晨报"→ 实时抓取 → 生成摘要 → 推送微信 → GitHub存档

**注意**: 手动触发不依赖预存文件，优先实时获取最新内容。

## 7. 已知问题与待优化

### 7.1 当前问题

| 问题 | 影响 | 状态 |
|------|------|------|
| 微信公众号 RSS 频繁 503 | 无法抓取公众号内容 | 待解决 |
| 知乎 API 403 Forbidden | 无法抓取知乎内容 | 待解决 |
| 永久去重导致内容递减 | 每天可推送内容变少 | 需确认用户需求 |
| arXiv 摘要中 "github.com" 误判 | 论文被错分为项目 | 已修复 |
| PhysiFormer 摘要无标点 | 句子分析失效 | 已修复（后备机制） |

### 7.2 待优化项

1. **引入 Kimi API**: 当前使用本地规则总结，虽然效果已大幅提升，但引入 Kimi API 可以生成更自然、更深入的摘要
2. **微信公众号代理**: 需要自建 RSS 代理或寻找稳定的第三方服务
3. **去重策略调整**: 考虑改为"按天去重"而非"永久去重"，或增加用户手动重置入口
4. **内容源扩展**: 增加 Papers With Code、Hugging Face 等源
5. **推送格式美化**: 支持 Markdown 格式、图片、表格等富文本
